## orchystraw - Bootstrap Prompt
## Give this to any AI (Claude Code, Windsurf, Codex) to set up multi-agent orchestration for ANY project.

You are setting up a multi-agent development orchestrator for this project.
Your job: analyze the codebase, determine the right agents, create all config files, and produce a working system.

---

## Step 1: Analyze the Project

Scan the codebase and determine:
1. **Languages & frameworks** - what's the tech stack?
2. **Directory structure** - what are the main code areas?
3. **Project type** - web app, mobile app, API, CLI, game, library, monorepo?
4. **Scale** - how many files? how complex?
5. **Existing CI/CD** - GitHub Actions? Other?
6. **Package manager** - npm, pip, cargo, swift, go mod?

Run these commands:
```bash
find . -name '*.ts' -o -name '*.tsx' -o -name '*.js' -o -name '*.jsx' -o -name '*.py' -o -name '*.swift' -o -name '*.rs' -o -name '*.go' -o -name '*.java' -o -name '*.kt' | head -100
wc -l $(find . -name '*.ts' -o -name '*.tsx' -o -name '*.py' -o -name '*.swift' -o -name '*.rs' -o -name '*.go' 2>/dev/null | head -50) 2>/dev/null | tail -1
ls -la
cat package.json 2>/dev/null | head -30 || cat Cargo.toml 2>/dev/null | head -20 || cat go.mod 2>/dev/null | head -10 || cat Package.swift 2>/dev/null | head -20 || cat requirements.txt 2>/dev/null | head -20
cat README.md 2>/dev/null | head -50
ls .github/workflows/ 2>/dev/null
```

### What to Include Per Agent

**Slash skills** (Claude Code built-ins - `/test`, `/review`, `/security`, `/debug`, `/refactor`):
- Backend agents: `/test` after new logic, `/review` after features, `/security` before auth code
- Frontend agents: `/test`, `/review`, `/debug`, `/refactor`
- Design agents: `/review`
- QA agents: ALL skills (`/security`, `/review`, `/test`, `/debug`)

**Superpowers plugin** (v4.3.0 - full workflow system, install via `/plugin install superpowers@superpowers-marketplace`):
- Core workflow: `brainstorming` = `writing-plans` = `subagent-driven-development` = `test-driven-development` = `verification-before-completion`
- Additional: `executing-plans`, `dispatching-parallel-agents`, `systematic-debugging`, `requesting-code-review`, `receiving-code-review`, `writing-skills`
- ⚠️ Do NOT use with auto-agent.sh: `using-git-worktrees`, `finishing-a-development-branch` (script manages git)
- Commands: `/brainstorm`, `/write-plan`, `/execute-plan`
- Per-agent: Backend gets TDD+SDD+debugging, Frontend gets SDD+plans, Design gets plans only, QA gets debugging+verification, PM gets brainstorming+plans
- Platform-specific: `supabase-backend-platform` (if Supabase), and other platform skills as available

**QMD** (local markdown search - reduces token cost for ALL agents):
- Install: `npm install -g @tobilu/qmd`
- Index: `qmd collection add docs --name project-docs && qmd collection add prompts --name project-prompts && qmd collection add src --name project-source && qmd embed`
- Add as MCP in `.mcp.json`: `{"qmd": {"command": "qmd", "args": ["mcp"]}}` - this makes it available to ALL agents
- Additionally install as PM agent skill (`.agents/skills/qmd/SKILL.md`) for prompt-gen searches
- **All agents benefit:** Backend searches API docs, Frontend searches design tokens, QA searches prior reports + findings, PM searches everything

**CRITICAL - Do NOT reference non-existent commands:**
- Valid slash commands: `/test`, `/review`, `/security`, `/debug`, `/refactor`, `/monitor`
- `/monitor` is defined in `~/Projects/CLAUDE.md` - tracks orchestrator cycles + background tasks
- Do NOT invent: `/plan`, `/deploy`, `/lint`, `/build` - these don't exist
- See `docs/SKILLS-REFERENCE.md` for the complete validated inventory

**MCPs** (from `.mcp.json` - project-specific external tools):
- Include MCP context in relevant agent prompts (e.g., Supabase MCP for backend, Context7 for all agents)
- QA should know about ALL MCPs to verify correct usage

**Agent Teams** (Claude Code sub-agent patterns):
- QA: `agent-teams:team-reviewer` (parallel multi-dimension review)
- Debug-heavy agents: `agent-teams:team-debugger`

**CLAUDE.md**:
- Reference it in EVERY agent prompt: "Read CLAUDE.md first - it has design system, code standards, anti-slop rules"
- Do NOT duplicate CLAUDE.md content in prompts - just reference it
- If CLAUDE.md doesn't exist, create one with: project overview, tech stack, code standards, design tokens

### 99-me Protocol (Human Escalation)
When any agent hits a blocker requiring the human to act:
- Append to `prompts/99-me/99-actions.txt` with status/priority/steps
- PM reviews each cycle
- Create `prompts/99-me/` directory during bootstrap

### PM Prompt Generation Workflow (5-Step Process)
PM follows this every cycle when updating agent prompts:
1. **Read current prompt** - what was assigned last cycle
2. **Review what got done** - git log, shared context, gh issue list, skim actual files
3. **Identify gaps & drift** - completed? partially done? not touched? emergent work?
4. **Update task sections only** - use Edit tool (not Write) to modify "What's DONE" and "YOUR TASKS" sections. Do NOT rewrite entire prompts. The orchestrator script auto-updates timestamps and file counts.
5. **Cross-check across agents** - no overlapping ownership, dependencies resolved

Include this workflow in the PM prompt you generate.

## Step 2: Design Agents

Based on your analysis, create agents. Rules:

### Mandatory Agents (EVERY project gets these 7)

These are non-negotiable. Every OrchyStraw project ships with:

| Agent | Role | Cycle | Key Rule |
|-------|------|-------|----------|
| **01-ceo** | Vision, strategy, market positioning | 3 | Sets direction, doesn't code |
| **02-cto** | Architecture, tech decisions, standards | 2 | Makes tech choices, reviews architecture |
| **03-pm** | Coordination, task assignment, prompts | 0 (LAST) | Never codes. Updates all other prompts |
| **XX-brand** | Naming, brand identity, voice | 3 | **RUNS FIRST on new projects. Nothing ships until name is locked.** |
| **XX-webdev** | Landing page + docs site | 2 | Owns `site/` — landing page + Mintlify docs |
| **XX-qa** | Quality assurance | 3-5 | Read-only. Reviews, doesn't fix |
| **XX-hr** | Team composition, onboarding, health | 3 | Manages agent team itself, not product |

### Brand Agent: NAMING IS FIRST PRIORITY

The brand agent's #1 job on any new project is **choosing the name**. Process:
1. Generate 50+ candidate names using 10 naming strategies (portmanteau, metaphor, invented, etc.)
2. Score each on 10 criteria (memorable, pronounceable, spellable, short, unique, domain, trademark, etc.)
3. Run `scripts/check-domain.sh` on top 10 — checks .com/.io/.app/.dev/.co/.ai + social handles
4. Also check: Canadian trademark (ic.gc.ca), US trademark (USPTO), App Store, NPM/PyPI
5. Present top 3-5 with full availability report
6. Once locked: update CLAUDE.md, brand guide, all agent prompts

**The brand agent blocks other agents until the name is decided.** No landing page, no docs, no marketing without a name.

### Domain Checker (built-in)
```bash
./scripts/check-domain.sh "name1" "name2" "name3"
# Checks: .com, .io, .app, .dev, .co, .ai + X + GitHub handles
# Uses whois (zero external deps)
```

### Worker Agents (2-4 additional, based on project type)

- **Agent boundaries must not overlap** - no two agents own the same directory.
- **Use `!path` exclusions** for fine-grained ownership (e.g., `src/ !src/components/ui/`).

## Step 3: Create All Files

### 3A: Create `scripts/agents.conf`

Format (pipe-delimited, no spaces around pipes):
```
# id | prompt_path | ownership | interval | label
02-backend | prompts/02-backend/02-backend-dev.txt | backend/ prisma/ | 1 | Backend Developer
03-frontend | prompts/03-frontend/03-frontend-dev.txt | frontend/ src/components/ | 1 | Frontend Developer
04-design | prompts/04-design/04-design-system.txt | src/styles/ src/components/ui/ !src/components/features/ | 1 | Design System
05-qa | prompts/05-qa/05-qa-review.txt | none | 5 | QA Engineer
```

Rules for ownership:
- Each path is space-separated within the ownership column
- Prefix with `!` to exclude (e.g., `src/ !src/styles/`)
- QA always has `none` (read-only reviewer)
- Never assign root dirs like `./` or `.` to any agent
- `prompts/` is always PM-owned (not in agents.conf)

## Lessons Learned

Follow these patterns.

### What Worked
1. **Shared context (00-shared-context/)** - agents reading each other's output eliminated duplicate work. Backend writes "added POST /batch", Frontend reads it and wires the endpoint same cycle.
2. **PM updating task sections every cycle** - prevents stale tasks, acknowledges shipped work, keeps agents focused on what's actually next. Script handles timestamps + file counts.
3. **File ownership enforcement** - the orchestrator commits files by ownership (not by agent). Backend can't accidentally commit frontend changes.
4. **99-me escalation** - agents appending to 99-me when blocked (env setup, API keys) means no silent failures. PM surfaces them.
5. **QA as final agent** - runs after workers, catches regressions same cycle.
6. **Session tracker as long-term memory** - task sections get updated each cycle, but SESSION_TRACKER.txt preserves: what shipped, decisions made, file counts. PM reads it to write accurate task assignments.
7. **Superpowers: writing-plans + verification-before-completion** - most impactful pair. Plans prevent scope creep, verification catches incomplete work.
8. **Agent-teams:team-implementer for big builds** - parallel sub-agents are faster than sequential for multi-file features.

### What Went Wrong (and How We Fixed It)
1. **Agent modified auto-agent.sh** - backend agent "improved" the orchestrator script. Fix: explicit "MUST NOT WRITE TO: scripts/" in every prompt.
2. **PM prompt grew to 400+ lines** - accumulated stale task lists. Fix: PM acknowledges done work then removes it. Target: 150-300 lines.
3. **Two agents edited same file** - two agents both touched the same component. Fix: use `!path` exclusions in agents.conf.
4. **Agent referenced nonexistent command** - prompt included commands that don't exist. Fix: validate every command against the real list before first cycle.
5. **No usage tracking** - early cycles had no budget awareness. Fix: `check-usage.sh` + `usage.txt` + PM reads it.
6. **QA findings not actionable** - "improve error handling" vs "auth.ts:142 - unwrapped API call, add try/catch". Fix: QA must cite file:line and severity for every finding.

### Scale Reference
- Works well with 3-5 agents for most projects
- Typical cost: ~$15-20 per full orchestrated cycle (varies by model and project size)
- PM prompt: 150-300 lines, Worker prompts: 170-380 lines each
