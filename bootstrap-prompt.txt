## Step 2: Design Agents

Based on your analysis, create 2-5 worker agents. Rules:

- **Every project gets a PM (01-pm)** â€” always. PM never codes.
- **Every project gets QA** â€” always. QA is read-only (ownership: none).
- **Workers are 2-4 agents** based on the code areas that need SEPARATE ownership.
- **Agent boundaries must not overlap** â€” no two agents own the same directory.
- **Use `!path` exclusions** for fine-grained ownership (e.g., `ios/ !ios/Components/`).

### Agent Design Patterns

| Project Type | Typical Agents |
|-------------|---------------|
| Full-stack web | 02-backend, 03-frontend, 04-design, 05-qa |
| iOS/Android app | 02-backend, 03-mobile, 04-design, 05-qa |

| Monorepo | 02-package-a, 03-package-b, 04-shared, 05-qa |
| CLI/library | 02-core, 03-docs, 04-qa |
| Game | 02-engine, 03-gameplay, 04-art-assets, 05-qa |

### Naming Convention
- `01-pm` â€” Project Manager (always)
- `02-XX` through `04-XX` â€” Workers (by domain)
- `05-qa` (or last number) â€” QA (always last, always read-only)

## Lessons Learned (battle-tested across 15+ auto-cycles, 5 agents, 441 files)

These are battle-tested patterns. Follow them.

### What Worked
1. **Shared context (00-shared-context/)** â€” agents reading each other's output eliminated duplicate work. Backend writes "added POST /batch", iOS reads it and wires the endpoint same cycle.
2. **PM updating task sections every cycle** â€” prevents stale tasks, acknowledges shipped work, keeps agents focused on what's actually next. Script handles timestamps + file counts.
3. **File ownership enforcement** â€” the orchestrator commits files by ownership (not by agent). Backend can't accidentally commit iOS changes.
4. **99-me escalation** â€” agents appending to 99-me when blocked (Xcode setup, API keys) means no silent failures. PM surfaces them.
5. **QA as final agent** â€” runs after workers, catches regressions same cycle. QA found 12 bugs in Cycle 1, all fixed by Cycle 2.
6. **Session tracker as long-term memory** â€” task sections get updated each cycle, but SESSION_TRACKER.txt preserves: what shipped, decisions made, file counts. PM reads it to write accurate task assignments.
7. **Superpowers: writing-plans + verification-before-completion** â€” most impactful pair. Plans prevent scope creep, verification catches incomplete work.
8. **Agent-teams:team-implementer for big builds** â€” 5Ă— faster than sequential for multi-file features.

### What Went Wrong (and How We Fixed It)
1. **Agent modified auto-agent.sh** â€” backend agent "improved" the orchestrator script. Fix: explicit "MUST NOT WRITE TO: scripts/" in every prompt.
2. **PM prompt grew to 400+ lines** â€” accumulated stale task lists. Fix: PM acknowledges done work then removes it. Target: 150-300 lines.
3. **Two agents edited same component** â€” iOS and Design both touched `ExpandableSection.swift`. Fix: use `!path` exclusions.
4. **Agent referenced nonexistent command** â€” `/monitor` appeared in all prompts (turned out to be valid from root CLAUDE.md, but could have been a hallucination). Fix: validate every command against the real list before first cycle.
5. **No usage tracking** â€” early cycles had no budget awareness. Fix: `check-usage.sh` + `usage.txt` + PM reads it.
6. **QA findings not actionable** â€” "improve error handling" vs "digests.ts:142 â€” extractActions Claude call unwrapped, add try/catch". Fix: QA must cite file:line and severity for every finding.

### Scale Reference (from production use)
- 226 source files (88 TS + 18 tests + 120 Swift)
- 302 tests, all passing
- 5 agents (PM + Backend + iOS + Design + QA)
- 3 auto-cycles in one day
- ~$15-20 total API cost across all cycles
