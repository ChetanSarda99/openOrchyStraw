## Bugs filed

### BUG-VQA-003 (MEDIUM): Landing site comparison table causes horizontal overflow on mobile

**Found in:** `site/src/components/comparison.tsx:145-148`
**Severity:** medium — visible on every mobile visitor, no console error but the whole page scrolls sideways
**Repro:**
```bash
cd site && npm run preview   # or manually serve out/ under /openOrchyStraw/
# open http://localhost:3737/openOrchyStraw/ in a mobile viewport (390×844)
```
Measured: `document.documentElement.scrollWidth === 656` at viewport 390 → **67% wider than the viewport**. Everything scrolls horizontally, not just the table.

**Root cause:**
- Line 148: `<table className="w-full min-w-[640px] border-collapse font-mono">` — the `min-w-[640px]` pins the table to 640px.
- Line 145: wrapper is `<motion.div className="mt-12 overflow-x-auto sm:mt-16">` — but **the wrapper itself grows to 640px**, defeating its own `overflow-x-auto`.
- The reason the wrapper grows: Tailwind flex/grid children have an intrinsic `min-width: auto`, so the wrapper does not shrink below its content width inside its `max-w-5xl` grid ancestor. Without `min-w-0` on the wrapper (or a parent), `overflow-x-auto` is effectively a no-op.
- Bbox confirmed: wrapper + table both at `left: 16, right: 656` at 390px viewport.

**Fix options (pick 1):**
- (a) Add `min-w-0` to the `motion.div` wrapper: `"mt-12 overflow-x-auto sm:mt-16 min-w-0"` (and likely its parent section inner container too). Lowest-risk fix, one line.
- (b) Replace table with a card layout on mobile (`sm:hidden` table, `block sm:hidden` card stack) — bigger change, better UX long-term.
- (c) Reduce `min-w-[640px]` to e.g. `min-w-[520px]` and ensure the overflow-x-auto scroll container clips properly — doesn't solve the root cause, just masks it.

**Assigned to:** 11-web (primary). 12-designer can weigh in on whether (a) or (b) is the right call for brand consistency — the comparison section is one of the most-read on the landing page.

### BUG-VQA-004 (LOW): App client router doesn't sync with URL — no deep linking, direct URL nav broken

**Found in:** `app/src/**` (React state-based navigation) interacting with `app/server.js:1086-1090` (SPA fallback)
**Severity:** low — functional for the common flow (single tab, click sidebar), but breaks refresh/bookmark/share/back-button expectations
**Repro:**
```bash
orchystraw app
# open http://127.0.0.1:4321/, click "Agents" in sidebar → Agents view renders ✓
# observe URL bar: still http://127.0.0.1:4321/  (not /agents)
# press browser back → nothing happens
# manually navigate to http://127.0.0.1:4321/agents → Dashboard renders, NOT Agents
# hard-refresh any non-root view → always lands on Dashboard
```
Verified via Playwright: clicking each of 7 sidebar items correctly switches content, but `page.url()` stays at `http://127.0.0.1:4321/` throughout.

**Root cause:**
- The sidebar uses React state (likely Zustand `activeView`) for navigation; it never calls `history.pushState`.
- `app/server.js` has a SPA fallback (`url === "/" ? "index.html" : url`) that serves `index.html` for non-file routes — so direct navigation to `/agents` gets the same HTML, but the client initializes `activeView` to `"dashboard"` regardless of `window.location.pathname`.
- Result: the SPA fallback in server.js is wired for a router that doesn't exist. Either the client should honor `pathname`, or the server fallback is dead code.

**Impact:**
- ❌ Can't deep-link to a specific view (e.g. share `/logs` with a teammate)
- ❌ Browser back/forward does nothing inside the app
- ❌ Refreshing `/logs` during a cycle run drops you back to Dashboard
- ❌ External links from Claude Code notifications can't land on a specific view
- ✅ The common "single session, click-driven" flow works fine, which is why this hasn't been noticed

**Fix options (pick 1):**
- (a) **Sync state ↔ URL via `history.pushState`** (lowest effort). Read `window.location.pathname` on mount to initialize `activeView`; on setActiveView, call `history.pushState({view}, '', '/' + view)`. Handle `popstate` to keep them in sync. ~30 lines.
- (b) **Adopt a real router** — React Router or wouter or tanstack-router. Higher effort but gives you typed routes + params.
- (c) **Declare state-based nav intentional** and delete `server.js:1086-1090` SPA fallback, returning 404 for non-root paths. Cleanest if the product is explicitly "no URL routing", but contradicts the current server behavior.

**Assigned to:** 06-backend (owns `app/server.js` + the state store). PM to decide between (a)/(b)/(c).
