## Findings

### [HIGH-01] `eval` on agent-controlled paths in `commit_by_ownership()`

**File:** `scripts/auto-agent.sh:236-241`
**Risk:** Command injection via crafted ownership paths in `agents.conf`

```bash
local changes=$(eval "git diff --name-only -- $include_paths $exclude_paths" ...)
eval "git add -- $include_paths $exclude_paths"
```

The `$include_paths` and `$exclude_paths` variables are built from `agents.conf` ownership fields. While `agents.conf` is currently protected (cannot be modified by agents), `eval` is inherently dangerous. If an agent ever found a way to influence these values (e.g., via a symlink attack or config corruption), arbitrary commands could execute.

**Severity:** HIGH
**Remediation:** Replace `eval` with array-based argument passing:
```bash
local -a git_args=(git diff --name-only --)
for path in $ownership; do
    [[ "$path" == !* ]] && git_args+=(":!${path#!}") || git_args+=("$path")
done
local changes=$("${git_args[@]}" 2>/dev/null | wc -l | tr -d ' ')
```
**Assign to:** 06-Backend

### [HIGH-02] `--dangerously-skip-permissions` grants full shell access to all agents

**File:** `scripts/auto-agent.sh:186, 472`
**Risk:** Any agent can execute arbitrary shell commands, read/write any file

This flag is required for autonomous operation, but it means the ownership boundaries in `agents.conf` are **prompt-enforced only** — not technically enforced. An agent that hallucinates or is prompt-injected could:
- Read credentials from `~/.claude/settings.json`
- Modify other agents' prompts
- Execute destructive commands

The `detect_rogue_writes()` function (lines 249-330) provides **post-hoc detection** which is good defense-in-depth, but cannot prevent data exfiltration or destructive commands that don't leave file traces.

**Severity:** HIGH (accepted risk, but document it)
**Remediation:**
1. Document this as an accepted risk in the threat model
2. Consider adding `--allowedTools` restrictions per agent when Claude Code supports it
3. The protected files list (lines 252-259) is a good mitigation — ensure it stays comprehensive
**Assign to:** 06-Backend (track), 02-CTO (architecture decision)

### [MEDIUM-01] `.gitignore` missing coverage for sensitive patterns

**File:** `.gitignore`
**Risk:** Accidental commit of secrets or credentials

Missing patterns:
- `.env`, `.env.*` — environment files
- `*.pem`, `*.key`, `*.p12` — certificates/keys
- `credentials.json`, `service-account*.json` — cloud credentials
- `.orchystraw/credentials` — referenced in docs as future credential store
- `prompts/00-backup/` — currently only covers `*.md`, not all backup files

**Severity:** MEDIUM
**Remediation:** Add to `.gitignore`:
```
.env
.env.*
*.pem
*.key
*.p12
credentials.json
service-account*.json
.orchystraw/credentials
```
**Assign to:** 06-Backend

### [MEDIUM-02] Notification function uses unescaped variable in PowerShell command

**File:** `scripts/auto-agent.sh:58-67`
**Risk:** Limited command injection via notification title

The `$escaped_title` variable is XML-escaped but not PowerShell-escaped. A title containing `$(...)` or backticks could execute PowerShell commands. Current risk is low because notification titles are generated by the script itself, not from agent output.

**Severity:** MEDIUM
**Remediation:** Wrap `$escaped_title` in single quotes within the PowerShell command, or sanitize for PowerShell metacharacters.
**Assign to:** 06-Backend

### [LOW-01] No lock file mechanism for concurrent orchestrator instances

**File:** `scripts/auto-agent.sh`
**Risk:** Two instances running simultaneously could cause git conflicts, race conditions

`.orchystraw.lock` is in `.gitignore` (suggesting it was planned) but not implemented in the script.

**Severity:** LOW
**Remediation:** Add lock file check at script start:
```bash
LOCK_FILE="$PROJECT_ROOT/.orchystraw.lock"
if [ -f "$LOCK_FILE" ] && kill -0 "$(cat "$LOCK_FILE")" 2>/dev/null; then
    echo "Another orchestrator is running (PID $(cat "$LOCK_FILE"))"
    exit 1
fi
echo $$ > "$LOCK_FILE"
trap "rm -f '$LOCK_FILE'" EXIT
```
**Assign to:** 06-Backend

---

### [LOW-02] `set -uo pipefail` missing `-e` (errexit)

**File:** `scripts/auto-agent.sh:23`
**Risk:** Script continues after errors that should halt execution

The script uses `set -uo pipefail` but omits `-e`. While the orchestrator intentionally continues past some failures (agent exits, git errors), missing `errexit` means unexpected errors in helper functions may silently propagate.

**Severity:** LOW (intentional design — orchestrator should be resilient)
**Remediation:** Consider adding `set -e` in helper functions that should fail-fast, while keeping the main loop resilient.
**Assign to:** 06-Backend

## Positive Security Observations

1. **Protected files mechanism** — `PROTECTED_FILES` array (line 252) prevents agents from modifying critical infrastructure. Well implemented with two-pass approach.
2. **Backup + validate cycle** — Prompts are backed up before PM runs and validated after. Corruption is detected and auto-restored.
3. **Rogue write detection** — Files outside all ownership boundaries are detected and reverted.
4. **Zero external dependencies** — Massive reduction in supply chain risk.
5. **Branch isolation** — Each cycle runs on a feature branch; merge conflicts don't corrupt main.
6. **Usage throttling** — Prevents runaway API costs with peak/off-peak awareness.
