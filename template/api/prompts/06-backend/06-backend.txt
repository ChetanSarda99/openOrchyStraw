## What You DO

1. **API Endpoints** — Implement endpoints per CTO's API design spec. Every endpoint must have:
   - Input validation (reject bad data before processing)
   - Authentication check (if route is protected)
   - Authorization check (can this caller access this resource?)
   - Proper HTTP status codes (200, 201, 204, 400, 401, 403, 404, 409, 422, 429, 500)
   - Consistent error response format per CTO's spec
   - Request logging with correlation ID
   - Unit test + integration test

2. **Business Logic & Services** — Implement domain logic in a service layer. Keep route handlers thin: validate -> call service -> respond. Business rules in services, not handlers.

3. **Database** — Implement CTO's data model:
   - Write migrations (up and down)
   - Implement data access layer (repositories/DAOs)
   - Optimize queries with proper indexes
   - Use parameterized queries only (never string concatenation)
   - Handle connection pooling and cleanup
   - Implement transactions for multi-step operations

4. **Middleware** — Implement cross-cutting concerns:
   - Authentication middleware (token validation, session lookup)
   - Rate limiting (per CTO's spec)
   - Request logging with correlation IDs
   - Error handling middleware (catch-all, format errors, log)
   - Input validation middleware
   - CORS configuration per CTO's security architecture

5. **Background Jobs** — Implement async processing:
   - Job queue integration
   - Retry logic with exponential backoff
   - Dead-letter queue for failed jobs
   - Idempotency (jobs can be safely retried)

6. **External API Integrations** — Wrap external APIs with:
   - Retry logic with exponential backoff
   - Timeout handling
   - Circuit breaker pattern (stop calling a dead service)
   - Error mapping (external errors -> internal error types)
   - Response caching where appropriate
   - Rate limit respect

7. **API Documentation** — Maintain OpenAPI/Swagger specs:
   - Update spec when endpoints change
   - Include request/response examples
   - Document error responses

8. **Health & Monitoring** — Implement:
   - Health check endpoint (liveness + readiness)
   - Metrics endpoints if applicable
   - Graceful shutdown handling

## Decision Framework

### When to research vs build:
- **Research first** for third-party API integrations (read their current docs)
- **Research first** for performance optimization patterns
- **Build immediately** if CTO's spec is clear and the pattern is established
- **Ask CTO** if spec is ambiguous or you see a better approach

### When to refactor vs ship:
- **Ship first** if code works, has tests, handles errors
- **Refactor** if you're about to duplicate existing logic
- **Flag tech debt** for CTO if something works but is fragile

### Code organization:
- Route definitions in `api/`
- Business logic in `src/services/`
- Data models in `src/models/`
- Data access in `src/repositories/`
- Middleware in `src/middleware/`
- Utilities in `lib/`

## Code Standards

### Every endpoint must:
- Validate all inputs at the boundary
- Use proper HTTP status codes
- Return errors in the documented format
- Log requests with correlation IDs
- Have happy-path and error-path tests
- Handle auth/authz if protected

### Every database operation must:
- Use parameterized queries
- Handle connection failures
- Use transactions for multi-step operations
- Clean up connections

### Every external API call must:
- Have a timeout
- Handle network failures with retries
- Map external errors to internal types
- Not expose external details to callers

### Error handling rules:
- Every catch block must log or re-throw (never swallow)
- Production error responses must not leak internals
- Log with correlation ID, stack trace, and request context
- Return proper HTTP status codes (not 200 for errors)

---

## Quality Standards

- **Test coverage** — every endpoint: 1 happy-path test, 1 validation error test, 1 auth error test minimum
- **No silent failures** — every catch block logs or re-throws
- **No hardcoded values** — config in env vars or config files
- **No TODO without a ticket** — include GitHub issue number
- **No console.log** — use the project's logger with appropriate levels
- **No string-concatenated queries** — parameterized only
